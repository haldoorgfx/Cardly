## TL;DR — CONFIRMED DATA-ISOLATION BREACHES (live-verified)

The mobile app relies **entirely on RLS**. Three additive "public read" policies (`public_attendee_wall` on `registrations`, `public_profile_read` on `profiles`, `public_read`/`is_visible` on `sponsors`) are written as **row-level** policies on tables that contain sensitive **columns**. RLS cannot restrict columns, so the *entire row* leaks to any anon/authenticated caller who queries the raw PostgREST endpoint (not just the display columns the app happens to select). I pulled real PII across tenants with the anon key.

| # | Table | What leaked (live) | Scope |
|---|-------|--------------------|-------|
| C1 | `registrations` | **79 rows across ALL events** — `attendee_email`, `attendee_phone`, `custom_fields` (arbitrary form PII), `amount_paid`, `payment_status`, `stripe_payment_intent_id`, `flutterwave_tx_ref`, and **`qr_code_token`** (the check-in credential) | Any confirmed/checked-in attendee of any public event |
| C2 | `profiles` | **Every user row** — `email`, `phone`, `plan`, `stripe_customer_id`, `role` (`super_admin` visible) | All public-identity users (organizers of published events + public-event attendees) |

## Root-cause theme

The single most damaging pattern: **using row-level `USING (…)` policies to grant "public read" on tables that mix public columns (name, avatar) with secret columns (email, phone, payment refs, QR tokens, invite tokens, Stripe IDs, admin role).** RLS is row-level; it cannot hide columns. The app "only selects display columns," but an attacker uses raw PostgREST `select=*`. Every such wall/profile/sponsor read should go through a **column-restricted view or `SECURITY DEFINER` RPC**, with direct `SELECT` on the base table revoked for anon/authenticated.

## Immediate actions (in order)
1. Kill the three public row policies (C1/C2/C3); replace with column-safe views/RPCs. **This stops the live PII + QR-token + admin-enumeration breach.**
2. Confirm migration `050` is applied on prod (poll_votes/qa_questions/attendee_agendas write lockdown). Apply if missing.
3. Authenticate/bind `/api/render` and `/api/v1/render` to the event owner / a real registration.
4. Fix `community_messages` insert check to require the caller owns the `registration_id`.
5. Add Stripe amount/currency assertion to `webhook` + `confirm-intent`.
6. Fix `teams`/`team_members` RLS recursion.
